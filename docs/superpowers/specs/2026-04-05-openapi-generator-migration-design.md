# Python Client Migration: swagger-codegen to openapi-generator

**Date:** 2026-04-05
**Scope:** Python client only; shared infrastructure updated to be tool-agnostic for future Go/JS migration

## Background

The Python client is currently generated by swagger-codegen 3.0.75 with 4 categories of post-generation patches and 22 custom Mustache templates. A spike with openapi-generator 7.21.0 showed that all patches become unnecessary, the output includes Pydantic v2 models with full type hints, and all 537 generated tests pass.

## Decisions

- **Approach B**: Migrate Python, make shared infra tool-agnostic, leave Go/JS on swagger-codegen
- **Templates**: Delete custom Mustache templates entirely (option A) — openapi-generator defaults are superior
- **Spike**: Delete spike output, do clean regeneration from updated scripts (option B)
- **Spec source**: Preserve existing behavior — default to GitHub URL, accept local path as CLI argument (option C)

## Design

### 1. `regen_common.py` Changes

Make the shared module work with both swagger-codegen (Go/JS) and openapi-generator (Python):

- New constant: `DEFAULT_OPENAPI_GENERATOR_CMD = "openapi-generator"`
- New prerequisite checker: `check_openapi_generator()` parallel to existing `check_swagger_codegen()`
- `run_codegen()` updated: accept a `tool` parameter (`"swagger-codegen"` or `"openapi-generator"`) controlling which CLI is invoked and which metadata directories are expected (`.swagger-codegen/` vs `.openapi-generator/`)
- Backup/restore logic: handle both `.swagger-codegen-ignore` and `.openapi-generator-ignore` based on tool parameter
- No breaking changes to Go/JS — they continue using the swagger-codegen tool parameter (or default)

### 2. `regenerate_python.py` Changes

- Switch CLI from `swagger-codegen generate -l python` to `openapi-generator generate -g python`
- Remove all post-generation patches:
  - Constructor `type`/`shape` parameter preservation (DfdDiagram, DfdDiagramInput, Edge, Node)
  - `auth_settings()` rewrite
  - Duplicate `revoke_token` method removal
  - `setup.py` Python version patching
- Remove custom templates reference (no `-t custom-templates/python/` flag)
- Delete `custom-templates/python/` directory (22 Mustache templates)
- Update metadata paths: `.swagger-codegen-ignore` -> `.openapi-generator-ignore`, `.swagger-codegen/` -> `.openapi-generator/`
- Update prerequisite check to `check_openapi_generator()`
- Rename config file to `openapi-generator-config.json` (content is compatible as-is)
- Update regeneration report to reference openapi-generator
- Keep: backup/restore for `pyproject.toml` and `test_diagram_fixes.py`, spec download, test execution, reporting

### 3. Integration Test Updates

`test_diagram_fixes.py` tests model instantiation and serialization. With Pydantic v2:

- Constructor call surface is the same (keyword arguments)
- Attribute access is the same (`.name`, `.type`)
- `swagger_types` and `attribute_map` class attributes no longer exist — remove references
- Serialization via `.to_dict()` still exists but output may differ slightly
- Pydantic enforces types strictly — fix any tests passing wrong types
- Preserve test intent: diagram instantiation, type preservation, serialization, inheritance, cell schemas

### 4. Cleanup and .gitignore

- Delete `python-client-generated/custom-templates/python/` directory
- Delete `spike-openapi-generator/` directory
- Add `.openapi-generator/` to `.gitignore`
- Keep `.swagger-codegen-ignore` references in Go/JS scripts untouched
- Regenerate Python client from scratch, run tests, verify clean output
